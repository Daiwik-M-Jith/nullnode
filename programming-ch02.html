Memory, Hardware, Algorithms & The "Hard" Stuff

1. The Hardware: How Computers Think

Before we talk about code, we must talk about the machine. Your computer is just a rock that we tricked into thinking by putting lightning inside it.

Binary & Hexadecimal

Computers only understand Voltage (On/Off). We represent this as 1 and 0 (Binary).

  • Bit: A single 1 or 0.
  • Byte: 8 bits (e.g., 10110011). Can represent numbers 0-255.
  • Hexadecimal (Hex): Binary is hard for humans to read. Hex is a shortcut. Instead of 0-9, we use 0-9 and A-F.
    1111 1111 (Binary) = 255 (Decimal) = FF (Hex).
    You'll see Hex everywhere in colors (#FFFFFF) and memory addresses (0x7fff5b).

The CPU Cycle

The CPU is the brain. It does three things, billions of times per second:

  1. Fetch: Get the next instruction from RAM.
  2. Decode: Figure out what the instruction means (e.g., "Add these two numbers").
  3. Execute: Do it.

2. Memory: The Stack vs The Heap

RAM isn't just a bucket. It's organized into two regions. Understanding this prevents crashes.

The Stack (Fast & Strict)

Analogy: A stack of plates. You can only touch the top one.

  • Storage: Local variables, function calls.
  • Speed: Instant. The CPU has a special register pointing to the top.
  • Limit: Small. If you recurse too deep, you get a Stack Overflow.
  • Cleanup: Automatic. When a function ends, its data is popped off.

The Heap (Big & Messy)

Analogy: A giant warehouse where you can put boxes anywhere.

  • Storage: Large objects, global data, things that need to live a long time.
  • Speed: Slower. You need a "Pointer" (address) to find your data.
  • Cleanup: Manual (C++) or Garbage Collected (Python/Java). If you forget to clean up, you get a Memory Leak.

What is a Pointer?

A pointer is just a variable that holds a memory address instead of a value.
Instead of holding the number 42, it holds 0x7fff5b (the location where 42 is stored).
Why? Passing a pointer to a 1GB video file is instant. Passing the file itself copies 1GB of data (slow).

3. Data Structures: The Toolbox

1. Arrays (The Egg Carton)

Fixed size, contiguous memory.
Good: Instant access (O(1)).
Bad: Can't grow easily. Inserting in the middle is slow (O(n)).

2. Linked Lists (The Treasure Hunt)

Nodes scattered in memory, each pointing to the next.
Good: Fast insertion/deletion (O(1)).
Bad: Slow access (O(n)) - you have to walk the chain.

3. Hash Maps (The Teleporter)

Key-Value pairs. Uses math to find location.
Good: Instant lookup (O(1)).
Bad: Unordered. Collisions can happen.

4. Trees (The Hierarchy)

Data with a parent-child relationship.
Example: Your computer's file system (Folders inside folders). The HTML DOM.
Binary Search Tree: A sorted tree where left is smaller, right is bigger. Allows super fast searching (O(log n)).

5. Graphs (The Network)

Nodes connected by edges. No hierarchy.
Example: Social Networks (Friends connected to friends). Google Maps (Cities connected by roads).

4. Algorithms: The Recipes

Big O Notation

Measuring how code slows down as data grows.
O(1): Instant.
O(log n): Fast (halving the problem).
O(n): Linear (reading a book).
O(n¬≤): Slow (nested loops).

Sorting Algorithms

Bubble Sort: Swapping neighbors repeatedly. Slow (O(n¬≤)).
Merge Sort: Splitting list in half, sorting, and merging. Fast (O(n log n)).
Why sort? You can't use Binary Search on unsorted data.

Recursion

A function that calls itself.
Used for traversing Trees and Graphs.
Danger: Infinite recursion = Stack Overflow.

5. Bitwise Operations: Hacker Math

Manipulating individual bits. This is how encryption and compression work.

AND (&)

Both must be 1.
1 & 1 = 1
1 & 0 = 0

OR (|)

Either can be 1.
1 | 0 = 1
0 | 0 = 0

XOR (^)

Must be different.
1 ^ 0 = 1
1 ^ 1 = 0

NOT (~)

Flip everything.
~1 = 0
~0 = 1

Fun Fact: XOR is used in encryption because it's reversible. (A ^ Key) ^ Key = A.

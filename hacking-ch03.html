üåê Topic 1: Linux Distributions ‚Äî Choosing Your Weapon

What ARE Linux Distributions?

Imagine Linux as a car engine (the kernel). A distribution is the entire car‚Äîcomplete with seats, dashboard, stereo system, and paint job. Different manufacturers (distro maintainers) take the same engine and build completely different vehicles for different purposes.

All Linux distros share the same core (the Linux kernel written by Linus Torvalds), but they differ in:

  • Package managers: How you install software (apt, yum, pacman)
  • Default software: Pre-installed tools and desktop environments
  • Release cycles: How often updates ship (rolling vs fixed)
  • Philosophy: Stability vs bleeding-edge features
  • Target audience: Beginners, hackers, servers, developers

The Big Families

üü¢ Debian Family (apt/dpkg)

Ubuntu: The beginner-friendly gateway drug to Linux. Massive community, tons of tutorials, hardware support out of the box. Perfect for learning basics before jumping into hacking-specific distros.

Debian: Ubuntu's parent. More stable, more minimal, less hand-holding. Favored for servers where you want rock-solid reliability.

Kali Linux: The legendary penetration testing distro. Pre-loaded with 600+ security tools. Built on Debian, optimized for hackers. This is where you'll live once you master the basics.

Package Manager: apt (Advanced Package Tool)
Example: sudo apt update && sudo apt install nmap

üî¥ Red Hat Family (yum/dnf)

Fedora: Bleeding-edge features, sponsored by Red Hat. Great for developers who want the latest toys. Updates frequently.

RHEL (Red Hat Enterprise Linux): Enterprise-grade, expensive, ultra-stable. Used in corporate environments.

CentOS: Free version of RHEL. Popular for servers and production environments.

Package Manager: dnf (or older yum)
Example: sudo dnf install wireshark

üîµ Arch Family (pacman)

Arch Linux: The DIY builder's dream. You construct your system from scratch. Rolling release means always-fresh packages. Steep learning curve but incredible control.

Manjaro: Arch made easy. Pre-configured for usability while keeping Arch's rolling release model.

BlackArch: Arch for penetration testing. Similar to Kali but Arch-based. Over 2,800+ security tools.

Package Manager: pacman
Example: sudo pacman -S metasploit

Which Distro Should YOU Use?

Total beginner? Start with Ubuntu or Linux Mint. Get comfortable with terminals, file systems, and package management.

Ready for hacking? Transition to Kali Linux or Parrot OS. Pre-loaded tools save you hours of setup.

Want to understand everything? Try Arch. You'll learn more in one Arch installation than months of using pre-built distros.

Pro tip: Use virtual machines (VirtualBox, VMware) to test multiple distros without commitment. Run Kali in a VM while keeping your main OS intact.

üî• Why This Matters for Hacking

Penetration testing requires specialized tools (Burp Suite, Metasploit, Aircrack-ng, SQLmap). Hacking distros like Kali come with these pre-installed and pre-configured. Trying to manually install 600+ tools on Windows or macOS? Good luck‚Äîyou'll spend weeks troubleshooting dependencies. With Kali, apt install and you're ready to exploit.

üìÇ Topic 2: Linux File System Hierarchy ‚Äî The Directory Structure Blueprint

Everything Starts at Root (/)

On Windows, you have drives: C:\, D:\, etc. On Linux, there's only one root: /

Think of / as the foundation of a building. Every directory branches off from here. No matter how many hard drives you plug in, they all mount somewhere under this single root.

The Sacred Directories

/bin ‚Äî Essential Binaries

Contains fundamental commands needed to boot and run the system. Commands like ls, cat, cp, mv, bash live here.

Hacker relevance: When you're in a minimal shell after exploiting a system, /bin commands will be your lifeline.

/etc ‚Äî Configuration Files

The system's brain. Every piece of software stores its configuration here. Network settings, user passwords (hashed), startup scripts, firewall rules.

Key files:

  • /etc/passwd ‚Äî User account info
  • /etc/shadow ‚Äî Password hashes (requires root)
  • /etc/hosts ‚Äî DNS mappings
  • /etc/ssh/sshd_config ‚Äî SSH server settings

Hacker relevance: After gaining access, /etc is where you hunt for misconfigurations, weak passwords, and privilege escalation paths.

/home ‚Äî User Directories

Each user gets a personal folder here: /home/username. This is where documents, downloads, and personal configs live.

Shortcut: ~ (tilde) always refers to your home directory. cd ~ takes you home instantly.

/var ‚Äî Variable Data

Data that changes constantly: logs, mail queues, databases, web server content.

Critical subdirectories:

  • /var/log ‚Äî System logs (auth.log, syslog, apache logs)
  • /var/www ‚Äî Web server root (nginx/apache)
  • /var/tmp ‚Äî Temp files that persist across reboots

Hacker relevance: Logs in /var/log reveal attack traces. Web roots often contain upload vulns. Always check here post-exploitation.

/tmp ‚Äî Temporary Files

A free-for-all directory where anyone can write. Cleared on reboot. Often world-writable.

Hacker relevance: Perfect place to drop payloads, compile exploits, or stage files. But defenders know this‚Äîexpect scrutiny.

/usr ‚Äî User Programs

Contains most user applications and utilities. Despite the name, this isn't user-specific‚Äîit's shared system-wide.

  • /usr/bin ‚Äî Non-essential commands (python, gcc, nmap)
  • /usr/sbin ‚Äî System admin tools
  • /usr/local ‚Äî Manually compiled software

/root ‚Äî Root User's Home

The superuser's personal directory. Not in /home‚Äîkept separate for security. Only root can access.

/proc and /sys ‚Äî Virtual File Systems

These don't exist on disk‚Äîthey're generated on-the-fly by the kernel. Represent running processes and system info.

Example: cat /proc/cpuinfo shows CPU details. /proc/[PID] directories contain info about each running process.

Navigating the File System

# Print working directory
pwd

# List files (long format, human-readable sizes, hidden files)
ls -lah

# Change directory
cd /etc

# Go up one level
cd ..

# Go to home directory
cd ~

# Go to previous directory
cd -

# Find files by name
find / -name "config.php" 2>/dev/null

# Search for text in files
grep -r "password" /etc 2>/dev/null

üî• Why This Matters for Hacking

When you compromise a Linux box, you need to navigate instantly. Knowing /etc/shadow has passwords, /var/log has evidence, and /tmp allows uploads saves precious time during an engagement. Defenders can't hide what you already know exists.

üîê Topic 3: Linux Permissions ‚Äî The Security Gatekeeper

Understanding the Permission Model

Imagine a nightclub with three VIP sections:

  • User (u): The owner‚Äîyou, the person who created the file
  • Group (g): Your crew‚Äîusers in the same group as the file
  • Others (o): Everyone else‚Äîrandom people on the system

Each section gets three types of access:

  • Read (r): View the file contents / List directory contents
  • Write (w): Modify the file / Add/delete files in directory
  • Execute (x): Run the file as a program / Enter the directory

Reading Permission Strings

$ ls -l important_file.txt
-rw-r--r-- 1 hacker hackers 4096 Dec 15 10:30 important_file.txt

Let's decode -rw-r--r--:

  • - ‚Äî File type (- = file, d = directory, l = symbolic link)
  • rw- ‚Äî User can read and write
  • r-- ‚Äî Group can only read
  • r-- ‚Äî Others can only read

Numeric (Octal) Notation

Permissions have numeric equivalents:

  • r (read) = 4
  • w (write) = 2
  • x (execute) = 1

Add them up:

  • 7 (rwx) = 4+2+1 = Full access
  • 6 (rw-) = 4+2 = Read and write
  • 5 (r-x) = 4+1 = Read and execute
  • 4 (r--) = 4 = Read only
  • 0 (---) = 0 = No access

Example: chmod 644 file.txt

  • 6 (rw-) ‚Äî User can read/write
  • 4 (r--) ‚Äî Group can read
  • 4 (r--) ‚Äî Others can read

chmod ‚Äî Change Mode (Permissions)

# Numeric method
chmod 755 script.sh   # rwxr-xr-x (common for executables)
chmod 600 secrets.txt # rw------- (private files)
chmod 777 public/     # rwxrwxrwx (DANGEROUS - everyone full access)

# Symbolic method
chmod u+x script.sh   # Add execute for user
chmod g-w file.txt    # Remove write from group
chmod o=r file.txt    # Set others to read-only
chmod a+r file.txt    # Add read for all (a = all)

# Recursive (for directories)
chmod -R 755 /var/www # Apply to all files inside

chown ‚Äî Change Owner

# Change owner
sudo chown alice file.txt

# Change owner and group
sudo chown alice:developers file.txt

# Recursive
sudo chown -R www-data:www-data /var/www/html

Special Permissions

SUID (Set User ID) ‚Äî 4000

When set on an executable, the file runs with the owner's permissions, not the executor's.

Example: /usr/bin/passwd has SUID. When you change your password, it temporarily runs as root to modify /etc/shadow.

Hacker gold: SUID binaries are prime privilege escalation targets. Find them: find / -perm -4000 2>/dev/null

SGID (Set Group ID) ‚Äî 2000

On files: Runs with group permissions. On directories: New files inherit the directory's group.

chmod 2755 shared_dir/

Sticky Bit ‚Äî 1000

On directories: Only the file owner can delete their files, even if others have write access to the directory.

Example: /tmp has the sticky bit. You can't delete someone else's files there.

chmod 1777 /tmp (represented by t in drwxrwxrwt)

umask ‚Äî Default Permissions

umask defines the default permissions for new files/directories by subtracting from maximum permissions.

  • Max for files: 666 (rw-rw-rw-)
  • Max for directories: 777 (rwxrwxrwx)

Example: If umask is 022:

  • New files: 666 - 022 = 644 (rw-r--r--)
  • New directories: 777 - 022 = 755 (rwxr-xr-x)

üî• Why This Matters for Hacking

Privilege escalation depends on finding misconfigurations. A world-writable config file (777)? You can modify it. A SUID binary you can exploit? Instant root. Defenders mess up permissions constantly‚Äîyour job is to find where. Commands like find / -writable 2>/dev/null become your best friend.

  • What does chmod 755 script.sh mean?
  • How do you find all SUID files on a system?
  • Why is chmod 777 dangerous?
  • What's the purpose of the sticky bit on /tmp?

üë• Topic 4: User Management & sudo ‚Äî Controlling Access

User Accounts

Linux is multi-user by design. Every process runs as a user. Every file belongs to a user. Understanding user management = understanding system security. 

# Create new user
sudo useradd -m -s /bin/bash alice

# Set password
sudo passwd alice

# Add user to group
sudo usermod -aG sudo alice  # Add to sudoers
sudo usermod -aG docker alice  # Add to docker group

# Delete user
sudo userdel -r alice  # -r removes home directory

# View user info
id alice
whoami
w  # Who's logged in
last  # Login history

The /etc/passwd File

root:x:0:0:root:/root:/bin/bash
alice:x:1001:1001:Alice Smith:/home/alice:/bin/bash

Format: username:password:UID:GID:comment:home:shell

  • password: x means password stored in /etc/shadow
  • UID 0: Root user (superuser)
  • UID 1-999: System accounts
  • UID 1000+: Regular users

The /etc/shadow File

Contains actual password hashes. Only readable by root. 

alice:$6$randomsalt$hashedpassword:19000:0:99999:7:::

sudo ‚Äî Superuser Do

sudo lets regular users execute commands as root (temporarily). 

# Run command as root
sudo apt update

# Open root shell
sudo -i  # Login shell
sudo -s  # Non-login shell

# Run as different user
sudo -u alice command

# Edit sudoers file (SAFELY)
sudo visudo  # Never edit /etc/sudoers directly!

# Check sudo privileges
sudo -l

sudoers Configuration

Example /etc/sudoers entries: 

# Allow alice to run any command
alice ALL=(ALL:ALL) ALL

# Allow developers group to run commands without password
%developers ALL=(ALL) NOPASSWD: ALL

# Allow specific commands only
bob ALL=(ALL) /usr/bin/apt, /usr/bin/systemctl

üî• Why This Matters for Hacking

Privilege escalation often starts with misconfigured sudo. sudo -l reveals what you can run as root. Find NOPASSWD entries? Gold mine. Exploitable binaries in sudoers? Root access. Always check sudo configs during penetration tests.

  • How do you add a user to the sudo group?
  • Where are password hashes stored?
  • What command shows your current sudo privileges?
  • Why is NOPASSWD in sudoers a security risk?

‚öôÔ∏è Topic 5: Process Management ‚Äî Controlling Running Programs

What IS a Process?

A process is a running instance of a program. When you execute ls, the kernel creates a process to run it.

Every process has a PID (Process ID) ‚Äî a unique number identifying it.

Viewing Processes

# List all processes
ps aux  # BSD style
ps -ef  # System V style

# Interactive process viewer
top
htop  # More user-friendly (install with apt/yum)

# Process tree (parent-child relationships)
pstree

# Find specific process
ps aux | grep apache2
pgrep apache2  # Returns PIDs only

Killing Processes

# Kill by PID
kill 1234

# Force kill (SIGKILL)
kill -9 1234

# Kill by name
killall firefox
pkill firefox

# Common signals
kill -15 1234  # SIGTERM (graceful shutdown)
kill -9 1234   # SIGKILL (immediate termination)
kill -1 1234   # SIGHUP (reload configuration)

Background & Foreground Jobs

# Run in background
command &

# Suspend current process (Ctrl+Z), then background it
bg

# Bring to foreground
fg

# List jobs
jobs

# Detach process from terminal (keeps running after logout)
nohup command &

systemd & Services

systemd is the modern init system managing services (daemons) on most Linux distributions. 

# Start/stop services
sudo systemctl start apache2
sudo systemctl stop apache2
sudo systemctl restart apache2
sudo systemctl reload apache2  # Reload config without restart

# Enable/disable autostart on boot
sudo systemctl enable apache2
sudo systemctl disable apache2

# Check service status
systemctl status apache2

# List all services
systemctl list-units --type=service

# View service logs
journalctl -u apache2 -f  # Follow logs live

üî• Why This Matters for Hacking

After exploiting a system, you need to understand what's running. ps aux reveals services you can target. Want persistence? Hijack a systemd service. Need to hide? Kill logging daemons. Process management is attack surface reconnaissance.

üéØ Quick Check

  • What's the difference between kill and kill -9?
  • How do you view all running services?
  • What command enables a service to start on boot?
  • How can you keep a process running after logging out?

üñ•Ô∏è Topic 7: Bash Scripting ‚Äî Automating Everything

Your First Script

#!/bin/bash
# This is a comment

echo "Hello, Hacker!"
echo "Current directory: $(pwd)"
echo "Current user: $(whoami)"

# Make executable and run
chmod +x script.sh
./script.sh

Variables

#!/bin/bash

# Define variables (no spaces around =)
name="Alice"
age=25

# Use variables
echo "Name: $name"
echo "Age: ${age}"

# Command substitution
current_date=$(date)
echo "Today is: $current_date"

# User input
read -p "Enter target IP: " target_ip
echo "Scanning $target_ip..."

Conditionals

#!/bin/bash

if [ -f "/etc/passwd" ]; then
    echo "/etc/passwd exists"
fi

if [ $USER == "root" ]; then
    echo "Running as root"
else
    echo "Not root user"
fi

# Numeric comparisons
if [ $age -gt 18 ]; then
    echo "Adult"
fi

Loops

#!/bin/bash

# For loop
for i in {1..5}; do
    echo "Iteration $i"
done

# Loop through files
for file in /etc/*.conf; do
    echo "Config file: $file"
done

# While loop
counter=1
while [ $counter -le 5 ]; do
    echo "Count: $counter"
    ((counter++))
done

Practical Hacking Script Example

#!/bin/bash
# Simple port scanner

if [ $# -eq 0 ]; then
    echo "Usage: $0 "
    exit 1
fi

target=$1
echo "Scanning $target for open ports..."

for port in {20..25} {80..443}; do
    timeout 1 bash -c "/dev/null
    if [ $? -eq 0 ]; then
        echo "Port $port is OPEN"
    fi
done

üî• Why This Matters for Hacking

Bash scripts automate reconnaissance, exploitation, and post-exploitation. Write scripts to scan networks, brute-force logins, enumerate users, exfiltrate data, maintain persistence. Every repetitive hacking task? Script it. Efficiency = more vulns found in less time.

üåê Topic 8: Networking & SSH ‚Äî Remote Access Mastery

Essential Networking Commands

# Network interfaces
ip addr show  # Modern command
ifconfig      # Legacy command

# Routing table
ip route show
route -n

# Active connections
netstat -tuln   # All listening ports
ss -tuln        # Modern alternative to netstat

# DNS lookup
nslookup example.com
dig example.com
host example.com

# Ping
ping -c 4 google.com

# Traceroute
traceroute google.com

SSH ‚Äî Secure Shell

SSH is your remote access lifeline. Encrypted terminal sessions to remote machines. 

# Connect to remote server
ssh user@192.168.1.100

# Specify port
ssh -p 2222 user@192.168.1.100

# Copy files (SCP)
scp file.txt user@server:/remote/path
scp user@server:/remote/file.txt .

# Copy directories
scp -r /local/dir user@server:/remote/path

# SSH key authentication (no password needed)
ssh-keygen -t rsa -b 4096
ssh-copy-id user@server
ssh user@server  # Now passwordless

SSH Configuration

Edit /etc/ssh/sshd_config for server settings: 

# Change port (security through obscurity)
Port 2222

# Disable root login
PermitRootLogin no

# Disable password authentication (key-only)
PasswordAuthentication no

# Restart SSH service after changes
sudo systemctl restart sshd

Firewall (iptables/ufw)

# UFW (Uncomplicated Firewall) - easier interface
sudo ufw status
sudo ufw enable
sudo ufw allow 22/tcp
sudo ufw allow 80/tcp
sudo ufw deny 23/tcp
sudo ufw delete allow 80/tcp

# iptables (more complex but powerful)
sudo iptables -L          # List rules
sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
sudo iptables -A INPUT -j DROP  # Drop all other traffic

üî• Why This Matters for Hacking

SSH is how you maintain access after exploitation. Weak SSH configs = easy entry. Look for default ports, password auth enabled, root login permitted. After gaining access, use SSH tunneling for pivoting. Understanding firewalls helps you bypass restrictions and exfiltrate data through allowed ports.

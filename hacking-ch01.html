Chapter 01

Computer Fundamentals

A complete beginner's guide to understanding how computers work at a fundamental level. This is the foundation you need before learning to hack anything.

Welcome to the beginning of your hacking journey. Before you can break into systems, exploit vulnerabilities, or write malicious code, you need to understand what you're actually attacking. This isn't optional knowledge‚Äîit's the foundation everything else is built on.

This chapter will teach you how computers actually work. Not the simplified version you learned in school, but the real, technical truth. By the end, you'll understand computers better than 95% of people using them every day.

Why This Matters for Hacking

Every hack exploits some fundamental aspect of how computers work. Buffer overflows exploit memory management. Code injection exploits how CPUs execute instructions. Privilege escalation exploits operating system design. If you don't understand the fundamentals, you're just copying code without knowing why it works.

What is a Computer? (The Real Answer)

Let's start with the most basic question: What actually is a computer? The answer is simpler than you think, but understanding it deeply will change how you see every digital device around you.

A computer is a machine that follows instructions. That's it. It doesn't "think," it doesn't "understand," and it has no intelligence whatsoever. It's a very fast, very obedient machine that does exactly what you tell it to do.

The Key Insight

Computers are deterministic machines. Given the same input and the same instructions, they will ALWAYS produce the same output. Always. This predictability is both their greatest strength and their biggest vulnerability.

When you "hack" a computer, you're not outsmarting an intelligent opponent. You're finding ways to give it instructions it wasn't supposed to receive, or making it misinterpret the instructions it has.

The Four Fundamental Parts

Every computer‚Äîfrom your phone to a supercomputer‚Äîhas exactly four fundamental components. Understanding these is critical because every attack targets one or more of them.

1. Input

What it is: Any way information enters the computer.

Examples: Keyboard, mouse, microphone, camera, network interface, USB devices, sensors, touchscreen.

Why hackers care: Input is where attacks begin. If you can control what data enters a system, you can often control the system itself. SQL injection, XSS, buffer overflows‚Äîthey all start with malicious input.

2. Processing (CPU)

What it is: The "brain" that executes instructions.

What it does: Math, logic, comparisons, moving data around. Despite being called a "brain," it's actually incredibly dumb‚Äîit just does these simple operations extremely fast (billions per second).

Why hackers care: The CPU doesn't care if instructions are legitimate or malicious. If you can inject your code into memory and make the CPU execute it, you own the system.

3. Storage/Memory

Two types:

‚Ä¢ RAM: Fast, temporary, volatile (clears when power off)

‚Ä¢ Disk: Slower, permanent, persistent

Why hackers care: Most attacks target RAM because that's where programs are actively running. Buffer overflows, heap exploits, code injection‚Äîthese all manipulate memory.

4. Output

What it is: Any way information leaves the computer.

Examples: Monitor, speakers, printer, network connection, LED lights.

Why hackers care: Output can leak sensitive data. Screen capture malware, network sniffing, keyloggers‚Äîthey intercept output. Even electromagnetic emissions can be captured (TEMPEST attacks).

The Hacker's Perspective

Every step in that process is a potential attack vector:

  • Input stage: USB Rubber Ducky attacks inject malicious keystrokes
  • Processing stage: Kernel exploits hijack interrupt handlers
  • Storage stage: Buffer overflows corrupt memory
  • Output stage: Screen capture malware steals display data

When we say computers are "fast," most people don't grasp the actual scale. Let's put it in perspective with real numbers.

The Restaurant Metaphor

Think of your computer like a restaurant kitchen:

  • CPU = The Chef: Fast, skilled, but can only work on one dish at a time (per core)
  • RAM = The Counter Space: Right next to the chef, holds ingredients for current orders
  • SSD = The Refrigerator: In the same room, takes a few seconds to walk to and grab ingredients
  • HDD = The Storage Room: Down the hallway, takes much longer to retrieve items
  • Internet = The Grocery Store: Across town, could take minutes or hours to get what you need

A good chef (CPU) keeps everything they need on the counter (RAM) so they don't waste time walking to the fridge (SSD) or storage room (HDD). That's why programs copy data from disk into RAM before using it!

How CPUs Actually Work

The CPU is called the "brain" of the computer, but that's misleading. It's not intelligent. It's a very fast calculator that follows instructions blindly. Understanding exactly how it works is critical for exploitation.

The Fetch-Execute Cycle

Every CPU in the world does the same basic process, billions of times per second. This cycle is so fundamental that if you understand it, you understand how all programs run.

// The Fetch-Execute Cycle (4 Steps): 1. FETCH // CPU reads the instruction pointer (IP/PC register) // This register holds a memory address instructionPointer = 0x00401000 // Example address instruction = readFromRAM(instructionPointer) // Gets instruction bytes from that address 2. DECODE // CPU's control unit interprets the instruction bytes // Determines what operation to perform bytes = "48 89 E5" // Example: Move RSP to RBP (x86-64) operation = decode(bytes) // Identifies which data/registers are needed 3. EXECUTE // CPU performs the actual operation if (operation == "ADD") -> performAddition() if (operation == "MOVE") -> moveData() if (operation == "JUMP") -> changeInstructionPointer() // Math, logic, data movement, or control flow 4. STORE // Write result back to memory or register writeResult(destination, result) instructionPointer += instructionSize // Move to next instruction updateFlags(zero, carry, overflow) // Then repeat. Forever. Until power off. while (computerIsOn) { fetch() decode() execute() store() }

The Recipe Metaphor

Imagine you're following a recipe to bake a cake. The fetch-execute cycle works exactly like this:

  • FETCH: Read the next line of the recipe ("Add 2 cups of flour")
  • DECODE: Understand what it means (need flour, need measuring cup, need mixing bowl)
  • EXECUTE: Actually do it (measure 2 cups, pour into bowl)
  • STORE: Update your state (bowl now has flour in it, move to next recipe line)

The CPU is like you following this recipe, but instead of baking a cake, it's executing program instructions. And instead of doing one step per minute, it does 3.5 BILLION steps per second!

Let's trace a real example with actual assembly code:

// Example: Calculate (5 + 3) and store result // Assembly Code: MOV EAX, 5 ; Move 5 into register EAX ADD EAX, 3 ; Add 3 to EAX MOV [result], EAX ; Store EAX to memory address "result" // Step-by-step execution: Instruction 1: MOV EAX, 5 FETCH: CPU reads instruction from memory (address 0x00401000) DECODE: "MOV register, immediate value" EXECUTE: Copy value 5 into EAX register STORE: EAX now contains 5, IP = 0x00401003 Instruction 2: ADD EAX, 3 FETCH: CPU reads instruction from memory (address 0x00401003) DECODE: "ADD register, immediate value" EXECUTE: ALU adds 3 to EAX (5 + 3 = 8) STORE: EAX now contains 8, IP = 0x00401006 Instruction 3: MOV [result], EAX FETCH: CPU reads instruction from memory (address 0x00401006) DECODE: "MOV memory address, register" EXECUTE: Copy EAX value to memory STORE: Memory address "result" now contains 8, IP = 0x0040100A // Total time on 3.5 GHz CPU: About 3 nanoseconds
üßÆ Let's Make This SUPER Clear - Adding Two Numbers

Imagine you're a first-grader and I ask you to add 5 + 3. Here's what your brain does:

  1. Read the problem: "5 + 3" (FETCH)
  2. Understand it: "Oh, I need to add two numbers" (DECODE)
  3. Do the math: "5... 6, 7, 8!" (EXECUTE)
  4. Write the answer: "8" on your paper (STORE)

That's EXACTLY what the CPU does! Except:

  • Instead of reading from paper, it reads from RAM
  • Instead of writing on paper, it writes to RAM or registers
  • Instead of taking 10 seconds, it takes 0.000000003 seconds (3 nanoseconds)

What is GHz? Why Does CPU Speed Matter?

You've seen specs like "3.5 GHz" or "4.2 GHz" when looking at CPUs. Here's what that actually means and why it's important for performance and security.

Understanding CPU Speed

Hz (Hertz): One cycle per second

MHz (Megahertz): One million cycles per second

GHz (Gigahertz): One billion cycles per second

A 3.5 GHz CPU completes 3.5 billion clock cycles every second. In most cases, one instruction takes one cycle, so that's roughly 3.5 billion instructions per second.

But modern CPUs are more complex. They use techniques like:

  • Pipelining: Start executing next instruction before current one finishes
  • Multiple cores: 4 cores = 4 instructions simultaneously
  • Hyper-threading: Each core pretends to be 2 cores
  • Out-of-order execution: Reorder instructions for efficiency
  • Branch prediction: Guess which code path will be taken

This complexity creates vulnerabilities. Spectre and Meltdown (famous CPU attacks from 2018) exploited speculative execution and branch prediction to leak secret data from memory.

Why This Matters for Hacking

The CPU is blind to code origin. It cannot tell the difference between:

  • Legitimate program code written by developers
  • Malicious shellcode injected by an attacker
  • Data that looks like code due to memory corruption

If you can place your instructions in memory and redirect the instruction pointer to them, the CPU will execute them. That's code execution‚Äîthe ultimate goal of many exploits.

CPU Registers: The Fastest Memory

CPUs have tiny, ultra-fast storage locations called registers. These are even faster than RAM and sit directly inside the CPU. Understanding registers is essential for reading assembly code and understanding exploits.

// Common x86-64 Registers (64-bit): General Purpose Registers (store data, do math): RAX - Accumulator (often holds return values) RBX - Base register RCX - Counter register (used in loops) RDX - Data register RSI - Source index (string/memory operations) RDI - Destination index (string/memory operations) R8-R15 - Additional general registers (x86-64 only) Special Purpose Registers: RIP - Instruction Pointer (address of next instruction) RSP - Stack Pointer (top of the stack) RBP - Base Pointer (bottom of current stack frame) RFLAGS - Status flags (zero, carry, overflow, etc.) // Why hackers care: // - Buffer overflows aim to overwrite RIP to hijack control // - Stack overflows corrupt RBP and RSP to crash or exploit // - Return-oriented programming chains gadgets using RAX, RDI, etc.

The Pocket Metaphor

Think of registers as the CPU's pockets. Imagine you're doing homework:

  • Your pockets (registers): Hold your pen, eraser, maybe a calculator - things you need RIGHT NOW
  • Your desk (RAM): Holds your textbook, notebook, papers - things you're actively using
  • Your backpack (SSD/HDD): Holds everything else - old homework, other books

Reaching into your pocket (register) is instant. Reaching across your desk (RAM) takes a moment. Opening your backpack (storage) takes much longer. The CPU works the same way!

üéÆ INTERACTIVE EXAMPLE: Register Usage in a Loop

Let's say we want to add up numbers from 1 to 5. Here's how registers are used:

// High-level code: int sum = 0; for (int i = 1; i <= 5; i++) { sum += i; } // What the CPU actually does (assembly): MOV RAX, 0 ; RAX = sum (starts at 0) MOV RCX, 1 ; RCX = i (counter starts at 1) loop_start: ADD RAX, RCX ; sum = sum + i INC RCX ; i++ (increment counter) CMP RCX, 5 ; Compare i with 5 JLE loop_start ; If i <= 5, jump back to loop_start // Final result: RAX contains 15 (1+2+3+4+5) // Let's trace it step by step: Iteration 1: RAX=0, RCX=1 ‚Üí RAX += 1 ‚Üí RAX=1, RCX=2 Iteration 2: RAX=1, RCX=2 ‚Üí RAX += 2 ‚Üí RAX=3, RCX=3 Iteration 3: RAX=3, RCX=3 ‚Üí RAX += 3 ‚Üí RAX=6, RCX=4 Iteration 4: RAX=6, RCX=4 ‚Üí RAX += 4 ‚Üí RAX=10, RCX=5 Iteration 5: RAX=10, RCX=5 ‚Üí RAX += 5 ‚Üí RAX=15, RCX=6 Loop exits: RCX > 5, so we stop // Total instructions executed: about 30 // Total time on 3.5 GHz CPU: about 8 nanoseconds

This is what EVERY for-loop becomes when your code runs. The CPU uses registers to keep track of everything - the sum, the counter, the limit. It's all just numbers being moved around and compared!

Memory Architecture: RAM vs Storage

This is where most confusion happens. People mix up RAM and storage constantly. Understanding the difference is absolutely critical for hacking because most exploits target memory (RAM), not storage.

The Fundamental Difference

Let's use an analogy that actually makes sense:

RAM = Your Desk Surface

Imagine your desk where you're working right now. The papers and tools you have spread out are like RAM:

  • Fast access: Just reach out and grab what you need
  • Limited space: Desk only holds so much
  • Temporary: Clear the desk at end of day
  • Active work: What you're using RIGHT NOW

When you need a new document, you pull it from the filing cabinet (storage) and put it on your desk (RAM).

The filing cabinet next to your desk is like storage (HDD/SSD):

  • Slow access: Get up, walk over, find folder, pull it out
  • Massive space: Can hold thousands of documents
  • Permanent: Files stay even when you go home
  • Long-term storage: Everything you MIGHT need later

Files stay in the cabinet until you need them. Then you copy them to your desk to work with them.

Attack Vectors by Memory Region

Stack attacks:

  • Buffer overflow: Write past array bounds, corrupt return address
  • Stack canary bypass: Defeat protection mechanisms
  • Format string: Use printf() bugs to read/write stack

Heap attacks:

  • Heap overflow: Corrupt heap metadata or adjacent allocations
  • Use-after-free: Access memory after it's been freed
  • Double-free: Free same memory twice, corrupt heap state

Code section attacks:

  • Return-oriented programming (ROP): Chain existing code snippets
  • Code cave injection: Find unused space in .text to hide code

Stack vs Heap: The Critical Difference

These two memory regions are used for different purposes and have completely different characteristics. Understanding both is essential. Let me explain this like you're 5 years old, then we'll get technical.

The Lego Metaphor (For Complete Beginners)

THE STACK = Building a Tower of Lego Blocks

  • You can only add blocks to the top (PUSH)
  • You can only remove blocks from the top (POP)
  • Very organized, very fast
  • Limited height (you'll run out of blocks)
  • Automatic: when you're done with a block, it goes back automatically

THE HEAP = A Messy Toy Box

  • You can put toys anywhere in the box
  • You can take toys out from anywhere
  • More flexible, but slower to find things
  • Much bigger box (more space)
  • Manual: YOU must remember to put toys back, or the box gets cluttered forever

The Stack (Technical Reality)

Purpose: Function calls and local variables

How it works:

  • Last-In-First-Out (LIFO) structure
  • Automatic memory management
  • Very fast (just pointer arithmetic)
  • Fixed size (typically 1-8 MB)
  • Grows downward in memory

What's stored:

  • Return addresses (where to go after function ends)
  • Function parameters
  • Local variables
  • Saved register values

Lifetime: Variables die when function returns

The Heap (Technical Reality)

Purpose: Dynamic memory allocation

How it works:

  • Random access structure
  • Manual memory management (you must free)
  • Slower (complex allocation algorithms)
  • Large size (limited by available RAM)
  • Grows upward in memory

What's stored:

  • Objects created with new/malloc
  • Large arrays or data structures
  • Data that must outlive function calls
  • Dynamically sized collections

Lifetime: Until you explicitly free it

Binary & Hexadecimal: The Language of Computers

Computers don't understand numbers the way humans do. They only understand electricity: ON or OFF. This creates a binary (base-2) system that's fundamental to everything in computing.

Why Binary?

At the physical level, computers use transistors‚Äîtiny switches that are either ON (1) or OFF (0). That's it. You can't have a transistor that's "kinda on" or "70% on." It's digital: binary.

Every piece of data in a computer‚Äînumbers, text, images, videos, programs‚Äîis ultimately represented as a sequence of 1s and 0s.

Understanding Bits and Bytes

Bit: A single binary digit (0 or 1). The smallest unit of data.

Byte: 8 bits grouped together. Can represent 256 different values (0-255).

Why 8? Historical reasons, but also practical: 8 bits is enough to represent all letters, numbers, and symbols (ASCII uses 7-8 bits per character).

Counting in Binary

Humans use base-10 (decimal) because we have 10 fingers. Computers use base-2 (binary) because they have 2 states. Let's see how counting works:

Decimal Binary Explanation
0 0000 All switches OFF
1 0001 Rightmost switch ON
2 0010 Next switch ON, carry over
3 0011 Both right switches ON
4 0100 Carry over again
5 0101 4 + 1
15 1111 All 4 switches ON
255 11111111 All 8 switches ON (1 byte max)

Hexadecimal: Binary Made Readable

Binary gets really long really fast. For example, the number 4,294,967,295 in binary is: 11111111111111111111111111111111 (32 digits!).

This is annoying for humans to read and write. So we use hexadecimal (base-16) as shorthand. Each hex digit represents 4 binary digits.

Why Do Computers Use Binary?

This is a question every beginner asks. The answer is surprisingly simple: physics and reliability.

‚ö° The Engineering Reality

Binary (2 states) is EASY to build and detect:

  • High voltage (5V) = 1
  • Low voltage (0V) = 0

With just two states, even if there's electrical noise or interference, it's easy to tell them apart. Is the voltage above 2.5V? It's a 1. Below? It's a 0. Simple!

Why not use 10 states (decimal)?

  • 0 = 0V, 1 = 0.5V, 2 = 1.0V, 3 = 1.5V... 9 = 4.5V
  • Problem: These voltages are too close together
  • Tiny electrical noise could turn a 4 into a 5
  • Reading would be slow and error-prone
  • Circuits would be much more complex

Binary = Fast, Reliable, Simple

Transistors are perfect for binary: they're either ON or OFF. This makes computers incredibly fast and reliable. Yes, it means numbers get long (11111111 instead of 255), but computers don't care about length‚Äîthey care about speed and accuracy.

Operating Systems: The Ultimate Control Program

The Operating System (OS) is the most important program on your computer. It's the boss, the manager, the referee. Every other program must ask the OS for permission to do anything. Understanding this hierarchy is CRITICAL for hacking.

What is an Operating System? (Really)

An OS is a program that manages all other programs. It's the intermediary between hardware (CPU, RAM, disk, network) and software (apps, games, browsers).

The Restaurant Metaphor

Think of your computer as a restaurant:

  • Hardware = The Kitchen: Stoves, ovens, refrigerators (CPU, RAM, disk)
  • Operating System = The Head Chef: Coordinates everything, assigns tasks, manages resources
  • Programs = Line Cooks: Each does a specific job (make salads, grill meat, bake desserts)
  • Users = Customers: Place orders, receive food

Line cooks (programs) can't just walk into the kitchen and use whatever they want. They must ask the head chef (OS) for permission. "Can I use the oven?" "Can I access the refrigerator?" The OS controls access to all resources.

The Four Core Jobs of an Operating System

1. Process Management

What: Running multiple programs at once

How:

  • Creates processes (running programs)
  • Schedules CPU time (time-slicing)
  • Switches between processes (context switching)
  • Terminates processes when done

You have 100+ processes running right now, but only a few CPU cores. The OS rapidly switches between them (thousands of times per second) to create the illusion of parallelism.

2. Memory Management

What: Allocating RAM to programs

How:

  • Allocates memory to each process
  • Isolates process memory (security!)
  • Implements virtual memory
  • Swaps to disk when RAM is full

Each program thinks it has all the RAM to itself. The OS creates this illusion using virtual memory, mapping virtual addresses to physical RAM.

3. File System Management

What: Organizing data on disk

How:

  • Creates/deletes files and folders
  • Reads/writes file data
  • Manages permissions
  • Maintains file metadata

Programs don't directly access the disk. They ask the OS: "Please read file X" and the OS handles the hardware details.

4. Device Management

What: Controlling hardware

How:

  • Communicates with hardware via drivers
  • Manages keyboard, mouse, display
  • Controls network cards, USB devices
  • Handles interrupts from hardware

Each piece of hardware has a driver (specialized software) that tells the OS how to talk to it. Programs don't touch hardware directly‚Äîthey go through the OS.

User Mode vs Kernel Mode: The Security Boundary

This is THE most important concept in OS security. Every exploit's goal is to cross this boundary.

üõ°Ô∏è THE SECURITY WALL
// Two execution modes in every CPU: USER MODE (Ring 3 - Least Privileged) // Where all normal programs run // Restricted access to: - Cannot directly access hardware - Cannot access other processes' memory - Cannot execute privileged CPU instructions - Cannot modify OS memory - Must ask OS for everything Examples: Chrome, Word, games, your code KERNEL MODE (Ring 0 - Full Privileges) // Where the OS kernel runs // Unrestricted access to: + Can directly control all hardware + Can access ANY memory address + Can execute ALL CPU instructions + Can modify anything, anywhere + Complete control of the computer Examples: Windows kernel, Linux kernel, device drivers // How programs cross the boundary: SYSTEM CALLS (syscalls) // Programs request OS services: open("/etc/passwd") ‚Üí "OS, please open this file" malloc(1024) ‚Üí "OS, please give me 1 KB of memory" send(socket, data) ‚Üí "OS, please send this over the network" fork() ‚Üí "OS, please create a new process" // Each syscall temporarily switches to kernel mode // OS validates the request, does the work, returns to user mode

Why Hackers Care: Privilege Escalation

The attacker's goal: Go from User Mode to Kernel Mode

When you initially compromise a system (via a web exploit, phishing, etc.), you're running in user mode with limited privileges. You can't:

  • Install rootkits or kernel backdoors
  • Read other users' files
  • Disable antivirus or security tools
  • Persist across reboots

Privilege escalation exploits find bugs in the OS kernel or drivers that allow user-mode code to execute kernel-mode code. Once you're in kernel mode, you're god‚Äîyou control everything.

File Systems: How Data is Actually Stored

When you "save a file," what actually happens? Where does it go? How does the computer find it again? File systems are the answer‚Äîand understanding them is crucial for finding hidden data, recovering deleted files, and hiding malware.

The File System's Job

Your disk is just billions of bytes. The file system organizes these bytes into files and folders, creating the structure you see in File Explorer/Finder.

What File Systems Do

  • Organize data: Create hierarchy (folders, subfolders, files)
  • Track locations: Remember where each file's bytes are on disk
  • Manage metadata: Store file names, sizes, dates, permissions
  • Allocate space: Find empty space for new files

Common File Systems

  • NTFS: Windows (supports permissions, encryption, large files)
  • FAT32: USB drives (simple, compatible, 4 GB file limit)
  • exFAT: Modern USB drives (no 4 GB limit)
  • ext4: Linux (journaling, reliability)
  • APFS: macOS (encryption, snapshots, SSD-optimized)

Networking Basics: How Computers Talk to Each Other

Everything you do online‚Äîbrowsing websites, sending messages, playing games‚Äîinvolves networking. Let me show you how data travels across the internet, from your computer to servers around the world.

The Simplest Explanation of the Internet

The internet is just millions of computers connected together, passing messages to each other. That's it. No magic, no mystery‚Äîjust data hopping from computer to computer until it reaches its destination.

The Postal Service Metaphor

Imagine sending a letter to someone across the country:

  • Your letter = Data packet: Contains the message and addresses
  • Your mailbox = Your computer: Where the letter starts
  • Post offices = Routers: Pass the letter along toward destination
  • Delivery route = Network path: Might go through many post offices
  • Recipient's mailbox = Destination server: Where letter ends up
  • Return address = IP address: So they can reply to you

Just like mail, internet data is broken into "packets" (like letters) that travel independently and might take different routes to reach the destination!

IP Addresses: The Internet's Phone Numbers

Every device on the internet needs a unique address so data knows where to go. That's an IP address.

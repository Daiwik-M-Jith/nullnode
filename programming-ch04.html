Line 1: Function Definition

def = Keyword meaning "define" - we're creating a function

fetch_user_data = Function name (snake_case convention in Python)

user_id = Parameter 1 (required argument)

timeout=10 = Parameter 2 with default value (optional, defaults to 10 if not provided)

: (colon) = Marks start of function body (everything indented below belongs to this function)

Line 2: Docstring (Documentation)

"""...""" (triple quotes) = Docstring - multi-line string explaining what function does

Purpose: Auto-generated documentation. Shows in IDE when you hover over function name.

Line 3: Import Statement

import = Keyword to bring in external code

requests = Library name (Python's HTTP library for making web requests)

Why import inside function? Lazy loading - only loads when function is called (saves memory)

Line 5-6: Variable Assignment & String Concatenation

base_url = Variable name (stores API base URL)

= (equals) = Assignment operator (not comparison! That's ==)

"..." (quotes) = String literal (text data)

+ (plus) = String concatenation (glue strings together)

str(user_id) = Type conversion (converts number to string so we can concatenate)

Result: If user_id=123, url becomes "https://api.example.com/users/123"

Line 8-11: Try Block (Happy Path)

try: = Start of error handling block (attempt risky code)

requests.get(url, timeout=timeout) = Function call breakdown:

  • requests = Module name
  • . (dot) = Member access (access function inside module)
  • get = Function name (sends HTTP GET request)
  • (url, timeout=timeout) = Arguments passed to function

response.raise_for_status() = Method call (if HTTP status 400/500, raise exception)

return response.json() = Return statement (exit function, send this value back to caller)

.json() = Method that converts HTTP response body to Python dictionary

Line 12-14: Exception Handling (Timeout)

except = Catch errors (if try block fails, run this instead of crashing)

requests.Timeout = Exception type to catch (specific to timeout errors)

f"..." = f-string (formatted string - lets you embed variables with {})

return None = Return special value meaning "no data"

Line 15-17: Exception Handling (HTTP Errors)

except requests.HTTPError as e = Catch specific exception AND give it a name

as = Alias keyword (call this exception e for short)

{e} = Insert exception details into string

Part 2: Common Code Patterns (You'll See These EVERYWHERE)

Pattern #1: List Comprehension (Compact Loop)

What it looks like:

// Old way (verbose)

squares = []

for i in range(10):

squares.append(i ** 2)

// New way (list comprehension)

squares = [i ** 2 for i in range(10)]

How to read it:

[expression for item in iterable]

  • expression = What to do with each item
  • item = Loop variable
  • iterable = Thing we're looping over

Why use it: Faster to write, easier to read once you're used to it. Very Pythonic.

Pattern #2: Ternary Operator (Inline If/Else)

// Old way

if age >= 18:

status = "adult"

else:

status = "minor"

// New way (ternary)

status = "adult" if age >= 18 else "minor"

How to read it:

value_if_true if condition else value_if_false

Pattern #3: Lambda Functions (Anonymous Functions)

// Normal function

def add(x, y):

return x + y

// Lambda (one-liner)

add = lambda x, y: x + y

Common use case:

// Sort list of tuples by second item

students = [("Alice", 85), ("Bob", 72), ("Charlie", 90)]

students.sort(key=lambda x: x[1])

# Result: [("Bob", 72), ("Alice", 85), ("Charlie", 90)]

Pattern #4: Tuple/List Unpacking

// Extract multiple values at once

x, y, z = (1, 2, 3)

# x=1, y=2, z=3

// Swap variables (no temp variable needed!)

a, b = b, a

// Unpack function return

name, age = get_user_info()

Pattern #5: Dictionary .get() with Default

// Bad (crashes if key doesn't exist)

value = mydict["key"] # KeyError if missing!

// Good (returns default if key missing)

value = mydict.get("key", "default_value")

Why important: Prevents crashes. If key doesn't exist, returns "default_value" instead of crashing.

Part 5: Debugging Like a Detective (Applying Ch03 Problem Decomposition)

From Chapter 03: You learned to break complex problems into simple pieces. Debugging = Break bug into pieces, test each piece, find the broken one. Same principle!

Strategy #1: Print Debugging (Most Common, Fastest)

The Idea: Insert print statements at strategic points to see what's happening. 

def calculate_discount(price, discount_percent):
    # Bug: Sometimes returns negative prices
    
    print(f"DEBUG: Input price={price}, discount={discount_percent}")
    
    discount_amount = price * (discount_percent / 100)
    print(f"DEBUG: Calculated discount_amount={discount_amount}")
    
    final_price = price - discount_amount
    print(f"DEBUG: Final price={final_price}")
    
    return final_price

# Run function with failing case:
calculate_discount(100, 150)  # 150% discount = negative price!

# Output shows the problem:
# DEBUG: Input price=100, discount=150
# DEBUG: Calculated discount_amount=150.0
DEBUG: Final price=-50.0  â† BUG FOUND!

Where to Place Prints:

  • Function entry: Print all input arguments
  • Before calculations: Print intermediate values
  • Inside loops: Print iteration number + values
  • Before return: Print final result

âš ï¸ Remember: DELETE or comment out debug prints before committing code!

Strategy #2: Breakpoint Debugging (IDE Debugger)

The Idea: Pause code execution at specific line, inspect variables, step through code line-by-line.

How to Use Breakpoints:

Step 1: Click left margin of line number â†’ Red dot appears (breakpoint set)

Step 2: Run code in debug mode (F5 or Debug button)

Step 3: Code PAUSES at breakpoint â†’ You can now:

  • Inspect variables: Hover over variable names to see values
  • Watch expressions: Add variables to "Watch" panel
  • Step Over (F10): Execute current line, move to next
  • Step Into (F11): Enter function call, debug inside it
  • Step Out (Shift+F11): Exit current function
  • Continue (F5): Resume until next breakpoint

When to use breakpoints instead of prints: Complex bugs with many variables, need to inspect object states, or step through loops one iteration at a time.

Strategy #3: Stack Trace Reading (Understanding Errors)

The Problem: Code crashes. You see 50 lines of error text. WHERE is the actual problem? 

Traceback (most recent call last):
  File "app.py", line 45, in <module>
    process_orders()
  File "app.py", line 38, in process_orders
    send_confirmation(order)
  File "email_service.py", line 12, in send_confirmation
    recipient = order["customer_email"]
KeyError: 'customer_email' â† THE ACTUAL ERROR!

How to Read Stack Traces (Bottom to Top):

1. Read LAST line first: KeyError: 'customer_email' = Dictionary missing this key

2. Find exact line: email_service.py, line 12 = Problem location

3. Read upward to understand path:

  • App started at line 45 (app.py)
  • Called process_orders() at line 38
  • Which called send_confirmation(order)
  • Which tried to access order["customer_email"] BUT it doesn't exist!

4. Root cause: order object doesn't have "customer_email" key. Fix in process_orders() before calling send_confirmation().

Strategy #4: Rubber Duck Debugging (Explain to Inanimate Object)

Sounds Crazy, Actually Works: Explain your code line-by-line to a rubber duck (or any object).

You (to duck): "Okay duck, first I get the user ID... then I query the database... then I calculate the total by multiplying price times quantity... wait, I'm using price but the database returns unit_price! THAT'S THE BUG!"

Why it works: Forcing yourself to explain code out loud catches logic errors your brain auto-corrects when reading silently.

No duck? Explain to coworker, friend, or even write explanation in comments. The act of explaining is what matters!

Strategy #5: Binary Search Debugging (Divide & Conquer from Ch03!)

From Chapter 03: Binary search = O(log n) efficiency. Works for bugs too!

Problem: Code has 1000 lines. Bug is SOMEWHERE in there. Checking line-by-line = O(n) = slow.

Solution (Binary Search Approach):

  • Step 1: Add print at line 500 (middle). Does bug occur before or after?
  • Step 2: If before: Add print at line 250. If after: Add print at line 750.
  • Step 3: Keep halving range until you pinpoint exact line.
  • Result: Found bug in ~10 checks instead of 1000! (logâ‚‚(1000) â‰ˆ 10)

This is Ch03 algorithms in action! Same O(log n) efficiency concept applied to debugging.

Step 2: Read Top-Down, Not Bottom-Up

WRONG: Reading every line from top to bottom.
RIGHT: Understand the big picture first, THEN dive into details.

1. Read the README - What does this code DO?

2. Check the imports - What libraries/modules does it use?

3. Scan function names - What are the main components?

4. Trace a specific feature - Follow one path through the code

5. Read detailed implementation - Now understand the HOW

Step 3: Understand Data Flow

Follow the data. Where does it come from? How does it transform? Where does it go?

# Example: User Login Flow

# 4. Password check (VERIFICATION)

6. Recognizing Bad Code (Code Smells)

"Code smell" = code that WORKS but is hard to read, maintain, or modify. Learning to spot these makes you a better developer.

Code Smell #1: Magic Numbers

send_reminder()

# GOOD: Ah, it's seconds in a day!

if time_elapsed > SECONDS_IN_DAY:

send_reminder()

Code Smell #2: God Functions (Too Long)

# BAD: 200-line function that does EVERYTHING

def process_order():

# ... 200 lines of code ...

# GOOD: Break into smaller functions

def process_order():

calculate_total()

charge_payment()

Code Smell #3: Deeply Nested Conditionals

# BAD: Pyramid of doom

if user.has_permission:

if resource.exists:

# Do stuff (4 levels deep!)

# GOOD: Early returns (guard clauses)

return error("No user")

if not user.is_authenticated:

return error("Not authenticated")

if not user.has_permission:

return error("No permission")

# Now do the happy path (flat!)

Refactoring Techniques

# Extract Method: Pull code into separate function

# BEFORE (BAD)

if item.discount:

# AFTER (GOOD)

subtotal = item.price * item.quantity

return subtotal - item.discount if item.discount else subtotal


total = sum(calculate_item_total(item) for item in cart)
